1. Executive Summary

Budget Flow is a web-based personal finance application that combines sign-up, onboarding, budgeting, savings tracking, reporting, currency-aware display, and AI-guided support in one user experience.

The current production environment is deployed on Render as a three-service architecture: a frontend web app, an authentication API, and a feature API backed by a shared PostgreSQL database. Groq is the current live AI provider, while Flowise remains optional.

Current live deployment
The application is currently reachable through the frontend at https://bf-web.onrender.com.

4. User Modules

Authentication

Users can register, sign in, sign in with Google, request a password reset, and complete password reset through a tokenized reset page.

Dashboard

The dashboard summarizes expenses, income, budget progress, and current-period overview data in a simple, readable layout.

Budgeting

Users can set a monthly budget, define budget categories, log or correct category spending, and edit or delete categories when needed.

Transactions

The transaction module supports entry creation, category assignment, date selection, history browsing, and protected deletion flows.

Savings

Users can create savings goals, define target amounts, add deposits, make withdrawals, and update or delete goals.

Reports and Assistant

The reporting module provides financial summaries, while the assistant uses live finance context to answer user questions through Groq or a safe local fallback.

5. Architecture

Frontend Layer

The frontend is split between the authentication experience and the main application experience. Runtime configuration allows the same frontend code to target local development, Render deployment, or alternative hosting without rebuilding the app for each environment.

Backend Layer

The platform uses two Bun + Express services. The auth backend owns identity-related flows, while the feature backend owns finance workflows, preferences, assistant behavior, and reporting.

Data Layer

PostgreSQL is the system of record for users, reset tokens, transactions, budgets, savings, profile settings, notification settings, and assistant-facing finance context.

Render Topology

The current Render release uses separate public services for the frontend and the two backend APIs, with one shared private Postgres instance. This keeps the UI lightweight while allowing the finance API and auth API to evolve independently.

Operational note
Free Render web services restrict outbound SMTP ports. For password reset at scale, a paid auth instance or an HTTP email API is the safer production path.

7. Security and Identity

  • JWT-based authentication is shared across the auth and feature services.
  • Google sign-in requires a matching `GOOGLE_CLIENT_ID` on the auth backend and the frontend origin to be listed in Google Cloud as an authorized JavaScript origin.
  • Password reset uses secure time-limited tokens and a reset flow tied to the configured client URL.
  • Irreversible actions such as deleting transactions, savings goals, categories, and accounts are protected with explicit confirmation UX.

9. Current Render Configuration

Frontend

The frontend service uses runtime env values so the browser can reach the live backend services without hardcoded domains.

Variable Value
AUTH_API_BASE_URL https://bf-auth-api.onrender.com/api/auth
FEATURE_API_BASE_URL https://bf-feature-api.onrender.com/api

Auth Backend

The auth service requires a live database URL, shared JWT secret, trusted frontend origin, Google client ID, and email provider configuration when password reset email is enabled.

Feature Backend

The feature service requires the same database, the same JWT secret as the auth backend, the live frontend origin, and Groq configuration. The current production setup keeps Flowise disabled.

11. Troubleshooting Guide

Frontend loads but actions fail

Check the frontend runtime configuration, backend origin values, and CORS-related environment variables.

Backend fails with ENOTFOUND

This usually means the database URL is wrong, copied from the wrong Render account, or the database and service are in different regions.

Google sign-in fails

Check the Google client ID, the authorized JavaScript origin list, and test-user access if the Google app is still in testing mode.

Password reset email fails

Check the email provider configuration and confirm the Render plan supports the chosen delivery method.
